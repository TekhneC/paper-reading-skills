# State Schema

This file documents derived workflow state stored inside this repository.

Zotero remains the bibliographic source of truth. State files here must not be treated as a replacement for Zotero metadata, and scripts must not modify Zotero tags, attachments, PDFs, or database records.

## `state/reading_queue.json`

Generated by:

```text
scripts/build_reading_queue_from_zotero_todo.py
```

Purpose: daily triage input derived from a configured Zotero todo tag or another approved queue source.

Top-level fields:

```text
schema_version           integer
generated_at             ISO-8601 UTC timestamp
source                   object describing queue source
daily_limit              integer or null
excluded_processed_count integer
paper_count              integer
papers                   array of paper records
```

Paper fields:

```text
paper_key                Zotero item key
zotero_item_id           Zotero itemID, for traceability only
library_id               Zotero libraryID
title                    paper title or "unknown"
item_type                Zotero item type
authors                  array of creator records
publication_title        venue/container title or null
date                     raw Zotero date field
year                     best-effort year
doi                      DOI or null
url                      URL or null
abstract_note            abstract text according to configured abstract policy
abstract_policy          full, truncate, or omit
abstract_available       boolean
tags                     Zotero tag names at generation time
date_added               Zotero dateAdded
date_modified            Zotero dateModified
status                   queued by default
quick_read_status        missing, available, generated, or unknown
pdf_status               available, missing, or unknown
priority_signals         array of queue source signals
notes                    null or short workflow note
```

The queue file may include truncated abstracts to support triage. If this repository will be committed or shared, prefer `abstract_policy = "truncate"` or `abstract_policy = "omit"` in `config/paths.toml`.

## `state/processed_papers.json`

Purpose: derived record used to avoid repeatedly selecting papers that still carry the Zotero todo tag.

Suggested shape:

```json
{
  "schema_version": 1,
  "updated_at": "YYYY-MM-DDTHH:MM:SSZ",
  "processed_paper_keys": [],
  "archived_paper_keys": [],
  "excluded_paper_keys": [],
  "papers": [
    {
      "paper_key": "<zotero key>",
      "title": "<paper title>",
      "status": "quick_read_done",
      "processed_at": "YYYY-MM-DDTHH:MM:SSZ",
      "source": "daily-paper-triage",
      "notes": null
    }
  ]
}
```

Queue-building scripts should exclude keys listed in:

```text
processed_paper_keys
archived_paper_keys
excluded_paper_keys
papers[] where status is quick_read_done, deep_read_candidate, deep_read_approved, deep_read_done, or archived
```

## `state/approvals/`

Purpose: explicit user approval records.

Only records in this directory, or direct user confirmation in the current interaction, may justify transition to:

```text
deep_read_approved
```

Recommended approval record path:

```text
state/approvals/<paper_key>.json
```

Approval conversion input:

```text
state/candidates/YYYY-MM-DD_deep_read_candidates.json
outputs/daily/YYYY-MM-DD/deep_read_candidates.json
paper_key
approver
approval_reason
approved_at
approved_focus
report_mode
```

Approval conversion output:

```json
{
  "schema_version": 1,
  "paper_key": "<zotero key>",
  "title": "<paper title>",
  "approval_status": "deep_read_approved",
  "approval_source": "user_confirmation | state_approval_record",
  "approved_at": "YYYY-MM-DDTHH:MM:SSZ",
  "approver": "<user or workflow name>",
  "source_candidate_file": "state/candidates/YYYY-MM-DD_deep_read_candidates.json",
  "approved_focus": "<deep-read focus>",
  "report_mode": "full_report | compact_report",
  "notes": null
}
```

Approval conversion must not create approval records for papers absent from the selected candidate file unless the current user request explicitly names the paper and approves deep reading.

## State transition policy

Allowed paper status values:

```text
queued
quick_read_done
deep_read_candidate
deep_read_approved
deep_read_done
archived
excluded
```

Allowed transitions:

```text
queued -> quick_read_done
quick_read_done -> deep_read_candidate
deep_read_candidate -> deep_read_approved
deep_read_approved -> deep_read_done
queued -> excluded
quick_read_done -> excluded
deep_read_candidate -> excluded
deep_read_done -> archived
```

Disallowed by default:

```text
queued -> deep_read_approved
queued -> deep_read_done
quick_read_done -> deep_read_done
deep_read_candidate -> deep_read_done
archived -> any active status
excluded -> any active status
```

Exceptions require explicit current-turn user approval and must be recorded in `notes`.

State update scripts must validate both the current status and the requested next status before writing. They must not silently invent missing intermediate states.

## Zotero todo tag cleanup

This workflow does not automatically remove, rename, or rewrite Zotero tags.

Recommended default behavior:

1. Keep Zotero unchanged.
2. Record completed or intentionally skipped papers in `state/processed_papers.json`.
3. Exclude processed keys when rebuilding `state/reading_queue.json`.
4. In the daily digest, propose a manual Zotero cleanup list for the user.

If automatic Zotero tag cleanup is ever added, it must be a separate explicitly approved workflow, not part of daily triage by default.

## Quick-read source cache

PDF-derived text may be written to:

```text
state/cache/quick_read_sources/<paper_key>.flow.txt
state/cache/quick_read_sources/<paper_key>.layout.txt
state/cache/quick_read_sources/<paper_key>.source.json
```

Purpose: local, derived, temporary evidence source for quick reading.

Rules:

```text
Do not commit this cache by default.
Do not treat cached extracted text as a reading report.
Use cached text only to support quick-read evidence extraction.
Keep Zotero PDFs and Zotero storage unchanged.
```

Source bundles produced by `scripts/extract_quick_read_source.py` should report:

```text
source_status.queue_entry_found
source_status.zotero_attachments_checked
source_status.pdf_available
source_status.pdf_text_extracted
pdf_sources[]
text_source.status
text_source.cache_path
text_source.text_mode
text_sources.flow.cache_path
text_sources.layout.cache_path
```

Default quick-read PDF extraction mode should be `both`:

```text
flow    Main reading order for prose.
layout  Tables, equations, and visual layout cross-checking.
```

For quick reading, use `flow` as the primary source and `layout` only to verify tables, metrics, baselines, ablation details, equations, or layout-sensitive claims.

Quick-read source caches, including `.flow.txt`, `.layout.txt`, and `.source.json`, should be treated as temporary. By default, cache files older than 7 days may be deleted by:

```text
scripts/extract_quick_read_source.py <paper_key> --cleanup-cache
```

Deep-reading workflows may use stronger extraction strategies such as page-level extraction, table extraction, or page images.

## Deep-read source cache

Deep-read source bundles may be written to:

```text
state/cache/deep_read_sources/<paper_key>/pages.json
state/cache/deep_read_sources/<paper_key>/tables.md
state/cache/deep_read_sources/<paper_key>/figures.json
state/cache/deep_read_sources/<paper_key>/figures/
state/cache/deep_read_sources/<paper_key>/source.json
```

Purpose: derived local evidence support for formal deep reading.

Recommended `pages.json` fields:

```text
paper_key
pdf_path
page_count
pages[].page_number
pages[].text
pages[].extraction_method
pages[].quality_flags
generated_at
```

Recommended `figures.json` fields:

```text
paper_key
selection_policy
key_figure_candidate
figures[].figure_id
figures[].page_number
figures[].caption
figures[].image_path
figures[].repo_relative_image_path
figures[].crop_image_path
figures[].repo_relative_crop_image_path
figures[].crop_bbox_points
figures[].crop_status
figures[].surrounding_text
figures[].role_hint
figures[].extraction_confidence
generated_at
```

Figure images are derived cache files. They may be embedded or linked from deep-read Markdown when needed to support interpretation, but original Zotero PDFs must remain untouched.

The first section of a deep-read report should include at most one key visual anchor when available:

```text
research article -> model architecture / method pipeline / system workflow
survey article -> scope diagram / taxonomy overview / review framework
default_or_unknown -> central process, framework, or scope figure
```

If no reliable key figure is available, the report should say that visual verification is needed instead of embedding an arbitrary figure.

When both full-page and cropped figure images are available, prefer `repo_relative_crop_image_path` for Markdown embedding. Use the full-page image only when the crop is missing, incorrect, or visually misleading.

Recommended `tables.md` format:

```text
## Table <n> / Page <n>

Caption:

Extracted text:

Reliability:
```

If table or figure extraction is unreliable, reports should mark the evidence as requiring page-level or visual verification rather than reconstructing results from corrupted text.

## Output validation

All skill outputs should be validated before they are treated as stable workflow state.

Validation targets:

```text
state/reading_queue.json
outputs/papers/<paper_key>/quick_read.json
outputs/daily/YYYY-MM-DD/quick_reads/<paper_key>.json
outputs/daily/YYYY-MM-DD/deep_read_candidates.json
state/approvals/<paper_key>.json
outputs/papers/<paper_key>/deep_read.json
outputs/daily/YYYY-MM-DD/deep_reads/<paper_key>.json
outputs/daily/YYYY-MM-DD/deep_reads.json
```

Validation should check:

```text
required fields exist
paper_key matches across paired md/json outputs
status values are known
state transitions are allowed
referenced files exist when required
approval exists before deep_read_done
deep_read Markdown and JSON sidecar both exist
external verification fields are explicit
```

## External verification records

Network verification may be needed for:

```text
preprint publication or acceptance status
DOI and publisher landing pages
OpenReview records
arXiv version history
adjacent surveys
author follow-up work
citing or follow-up works
project pages
```

Verification records may be written to:

```text
state/external_verification/<paper_key>.json
```

Recommended fields:

```text
schema_version
paper_key
verified_at
verification_status = verified | partially_verified | unverified | skipped
queries[]
sources[].source_type
sources[].title
sources[].url
sources[].checked_at
sources[].finding
sources[].confidence
notes
```

When network access is unavailable, record `verification_status = unverified` or `skipped`; do not fabricate external findings.
